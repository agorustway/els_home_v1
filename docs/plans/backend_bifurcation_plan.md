# 🚀 ELS Backend Bifurcation Plan (v4.5.0)
> **목표**: 봇 업데이트(40분 빌드) 중에도 차량 관제 및 배차판 자동 동기화 서비스 상각(High Availability) 확보

## 📐 신규 아키텍처 (Master Blueprint)
기존 하나의 무거운 컨테이너를 **[ELS-BOT]**과 **[ELS-CORE-API]**로 이원화합니다.

| 서비스명 | 역할 | 베이스 이미지 | 빌드 속도 | 주요 기능 |
| :--- | :--- | :--- | :--- | :--- |
| **els-bot** | Selenium 엔진 | `python:3.9-slim` + Chrome | 🐌 느림 (40분+) | 컨테이너 이력 조회, 세션 유지 |
| **els-core** | 경량 API 서버 | `python:3.9-slim` | ⚡ 빠름 (1분 내) | 차량 관제, 배차판 동기화, 로그 저장 |

---

## 🛠️ PDCA 단계별 구축 로직 (Step-by-Step)

### 1️⃣ PLAN (설계)
- [ ] `docker-compose.yml`에서 서비스 분리 및 포트 할당 정의
  - `els-bot`: 2929 (기존 유지)
  - `els-core`: 2930 (신설)
- [ ] 공유 볼륨(`./data`)을 두 컨테이너에 공통 마운트하여 데이터 정합성 유지

### 2️⃣ DO (구현)
- [ ] `docker/els-backend/app.py`에서 봇 관련 로직과 API 로직 분리
  - `app_bot.py`: 로그인, 조회, 데몬 제어 전용
  - `app_core.py`: 관제, 스케줄러, 로그 전용
- [ ] `Dockerfile`을 경량화 버전(API용)과 풀셋 버전(BOT용)으로 이원화
- [ ] 프론트엔드(`web/`)의 API 호출 주소를 용도에 맞게 분산

### 3️⃣ CHECK (검증 - TDD)
- [ ] `py -m py_compile`로 신규 소스코드 문법 완벽 검토
- [ ] 로컬 환경에서 두 백엔드가 동시에 수파베이스에 정상 접속하는지 확인
- [ ] 봇 빌드 도중 API 서버가 독자적으로 통신되는지 흉내 테스트(Dry Run)

### 4️⃣ ACT (배포 및 최적화)
- [ ] NAS에 점진적 배포 (API 먼저 띄우고 봇 빌드 시작)
- [ ] `docs/01_MISSION_CONTROL.md` 최신화 및 안정성 모니터링

## ⚠️ 리스크 관리 (Safety First)
- **포트 충돌**: 시놀로지 NAS의 다른 포트(8443 등)와 겹치지 않도록 신규 포트(2930) 예약 확인 필수.
- **Vercel 설정**: 유저가 사용하는 웹에서의 API 엔드포인트를 두 갈래로 찢어야 하므로, Vercel 환경 변수 설계 교정 필요.
